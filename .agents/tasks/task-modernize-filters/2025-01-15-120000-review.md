# Filter modernization and bill statements client-side filtering

This change strips the neo-brutalist visual treatment (thick borders, offset box-shadows, bold weight overrides) from the expenses filter and page card, replaces it with standard shadcn/ui component styling, and adds date-range presets (Today, Last 7/30 days, This/Last month) to the date picker popover. A new `BillStatementsFilters` component provides client-side search, status, and date-range filtering for the bill statements page, with stats cards recomputing against the filtered set.

Watch for: the bill statements filter triggers `onFilteredChange` on every keystroke via `useEffect`, which can cause render cascades on large lists (likely). The "Total Statements" stat card still reads from the unfiltered `billStatements` array while the other three read from `filteredBillStatements`, creating an inconsistency in what the stats communicate when filters are active (confirmed). The date picker preset sidebar has no visual feedback indicating which preset is currently active (confirmed).

## High-level view

The expenses filter rewrites its container and controls to use the default border/shadow from shadcn primitives, replaces the custom `neoControlClass`/`neoButtonClass`/`neoPopoverClass` variables with nothing, and restructures the layout into a two-row grid: primary filters on top, date/sort/actions below. The date popover gains a left sidebar of quick-select buttons that call `handlePresetSelect` to set the range state directly.

The bill statements filter is a new component that owns local filter state and pushes filtered results to its parent via `onFilteredChange`. It applies filters client-side in a `useEffect` that runs on every state change, producing immediate visual feedback without an Apply button. The page integrates it by threading a `filteredBillStatements` state through the stats and table, with a sync effect to reset when the source data changes.

The two filter components share an identical date-picker-with-presets block (~90 lines) that is not extracted into a shared component. This is a maintenance risk if the preset list or layout changes in one place but not the other.

<summary>Issues (5)</summary>

1. **Stats inconsistency when filters active** (confirmed) — "Total Statements" shows the unfiltered count while Active/Inactive/Overdue show filtered counts. Either all stats should reflect the filtered set or "Total Statements" should be labeled "Total (unfiltered)" to avoid misleading the user.
2. **Render cascade on every keystroke** (likely) — `BillStatementsFilters` calls `onFilteredChange(filtered)` inside a `useEffect` triggered by `search` state. Each keystroke re-filters and calls `setFilteredBillStatements` in the parent, re-rendering stats + table. Add a debounce on the search input (200-300ms) or move to an event-driven apply model.
3. **No active-preset indicator** (confirmed) — The date presets sidebar does not highlight which preset is currently selected. After clicking "Last 7 days", all buttons still look identical. Track the active preset key in state and apply a visual indicator.
4. **Duplicated date picker block** (confirmed) — The preset sidebar + calendar popover is copy-pasted identically between `expenses-filters.tsx` and `bill-statements-filters.tsx`. Extract a `DateRangePickerWithPresets` component to avoid drift.
5. **`onFilteredChange` stability assumption** (possible) — The page passes `setFilteredBillStatements` directly (stable because it's a state setter), but if the prop ever becomes an inline callback the effect would re-run on every parent render. Consider wrapping the effect's use of the callback in a ref to decouple it from the dependency array.

## Stats card asymmetry on the bill statements page

The "Total Statements" card references `billStatements.length` while Active, Inactive, and Overdue reference `filteredBillStatements`. When a user applies a filter, the total stays at the original count while the sub-counts shrink:

```tsx
// Total — reads unfiltered
{isLoading ? "-" : billStatements.length}

// Active — reads filtered
{filteredBillStatements.filter((bs) => bs.is_active).length}
```

A user sees "Total: 12, Active: 3, Inactive: 1" and wonders where the other 8 went. Either change "Total Statements" to read `filteredBillStatements.length`, or show "showing X of Y".

## Filter-on-every-keystroke without debounce

`BillStatementsFilters` runs its filtering `useEffect` whenever `search` changes. Since `setSearch` fires on every `onChange`, typing "electricity" triggers 11 filter + re-render cycles. For small lists this is imperceptible, but the pattern encourages the same approach on heavier datasets without any throttle. A 200ms debounce on the search value fed to the effect would prevent intermediate renders while preserving the immediate-apply UX.

The expenses filter sidesteps this entirely by requiring an explicit Apply click — the two pages present different interaction models for text search.

## Duplicated date picker with presets

Both filter components contain an identical ~90-line block: a `Popover` wrapping a flex layout with 5 preset buttons (computing dates via `date-fns`) and a `Calendar` in range mode. The only difference is the trigger button width (`w-full` vs `w-full sm:w-[260px]`). If a sixth preset is added, or the layout is adjusted for mobile (presets stacked below), the change must be made in two places. Extracting a `DateRangePickerWithPresets` component with a configurable trigger width would eliminate this.

## No visual indicator for active date preset

After clicking a preset button, the calendar selection updates but every preset button remains `variant="ghost"` with no highlight. There's no state tracking which preset was last selected. Since the user can also drag a custom range on the calendar that won't correspond to any preset, the cleanest approach is to store the active preset key in state and clear it when `onSelect` fires from the calendar directly.
